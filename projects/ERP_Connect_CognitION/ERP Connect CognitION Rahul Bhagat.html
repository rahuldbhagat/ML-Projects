Technology

Tech Stack & Significance

Open-source, cost-effective technologies powering ERP Connect CognitION.

Technology Category Significance
DistilGPT-2 Base Model Lightweight (82M params), fast inference, low compute requirements. Ideal for fine-tuning on limited resources like free Colab GPU.
LoRA Fine-Tuning Method Reduces trainable parameters from 82M to ~0.5M. Enables efficient domain adaptation without full model retraining, saving time and cost.
PEFT Training Library Hugging Face library that implements LoRA. Simplifies adapter-based training and checkpoint management.
Hugging Face Transformers ML Framework Industry-standard library for loading, training, and deploying LLMs. Provides Trainer API for streamlined training loops.
ChromaDB Vector Database Lightweight, open-source vector store for RAG. Stores FAQ embeddings for fast similarity search at inference time.
SentenceTransformers Embedding Model Generates semantic embeddings (all-MiniLM-L6-v2) for queries and FAQs. Enables accurate context retrieval in RAG pipeline.
RAG Architecture Pattern Grounds model responses in real FAQ data, reducing hallucinations. Allows knowledge updates without retraining.
Gradio UI Framework Rapid prototyping of web-based demo interface. Enables non-technical stakeholders to test the model easily.
Google Colab (T4 GPU) Compute Platform Free GPU access for training. Democratizes LLM fine-tuning without infrastructure investment.
JSONL Data Format Standard format for training data. Each line is a self-contained JSON object, enabling streaming and easy validation.
BLEU / ROUGE-L Evaluation Metrics Automated metrics for measuring response quality against reference answers. Provides quantitative benchmarks.
Cosine Similarity Evaluation Metric Measures semantic similarity between generated and expected responses using embeddings. Captures meaning beyond exact word matches.

Transparency

Assumptions & Limitations

Important factors to consider for successful implementation.

✓ Key Assumptions

  • 📊
    Quality Dataset Available We can synthesize or collect ~1000 high-quality Q&A pairs from existing support tickets and documentation.
  • 💻
    Minimal Compute Requirements LoRA fine-tuning can be done on Google Colab (free T4 GPU) or equivalent low-cost infrastructure.
  • 🔄
    Iterative Improvement The model can be continuously improved with user feedback and new FAQ entries.
  • 👥
    Human Oversight Support staff will review and validate AI responses before they reach production use.

⚠ Known Limitations

  • 🧠
    Model Size Constraints DistilGPT-2 (82M params) may struggle with highly complex, multi-step troubleshooting scenarios.
  • 📝
    Training Data Dependency Response quality directly depends on the quality and coverage of the FAQ dataset.
  • 🆕
    Novel Issues The model cannot handle completely new error types not present in training data without RAG fallback.
  • 🔒
    No Real-time Data Access The model cannot query live systems; it provides guidance based on static knowledge.
Roadmap

Implementation Plan

Phased approach to deliver value quickly while building toward full capability.

1

Dataset Creation & Validation Week 1-2

Compile FAQ corpus from support tickets, documentation, and SME interviews. Clean, format, and validate ~1000 Q&A pairs in JSONL format.

2

Model Fine-Tuning Week 2-3

Set up LoRA training pipeline on Colab. Fine-tune DistilGPT-2 with early stopping. Generate model checkpoints and validate loss curves.

3

RAG Pipeline Integration Week 3

Build vector index with ChromaDB. Implement retrieval pipeline with SentenceTransformers. Connect RAG to fine-tuned model.

4

Evaluation & Testing Week 4

Run automated metrics (BLEU, ROUGE, cosine similarity). Conduct manual evaluation on 50+ test queries. Compare base vs fine-tuned performance.

5

Demo & Documentation Week 4-5

Build Gradio interface for stakeholder demos. Create comprehensive README with training details, results, and usage instructions.

6

Pilot Deployment Week 6+

Deploy to internal support team for beta testing. Collect feedback, iterate on model. Plan production integration with existing support systems.

User Query 1
"Error Server responded with HTTP 403?"
AI Response
HTTP 403 (Forbidden) means the server understood the request but refuses to authorize it. In simple terms, you reached the server successfully but you don't have permission to access that resource. Typical reasons in ERP-Connect integration: • Missing or invalid API Gateway authorization token • User role restrictions in SAP (check PFCG assignments) • IP allowlist/firewall rules blocking the request • Resource access policies in ECS task role Key difference from 401: 401 means "not authenticated" (login required), while 403 means "authenticated but not allowed." One-liner: "I know who you are, but you can't access this." Next steps: Check API Gateway authorizer logs, verify IAM roles, and confirm SAP user permissions.
User Query 2
"BAPI_GOODSMVT_CREATE returned error: Material document not posted"
AI Response
This error occurs when SAP's BAPI_GOODSMVT_CREATE fails to post a goods movement document. The BAPI performs validation checks before committing the material document to the database. Common root causes: • Missing or invalid Movement Type (BWART) in the GM_CODE parameter • Material master not extended to the target plant/storage location • Insufficient stock for goods issue (Movement Type 261/262) • Missing batch determination for batch-managed materials • Posting period not open in MM Period (MMPV transaction) Debugging steps: 1. Check the RETURN table structure for detailed error messages (MESSAGE_V1 to MESSAGE_V4) 2. Verify GOODSMVT_HEADER fields: DOC_DATE, PSTNG_DATE, PR_UNAME 3. Validate GOODSMVT_ITEM entries: MATERIAL, PLANT, STGE_LOC, MOVE_TYPE, ENTRY_QNT Resolution: Use transaction SE37 to test BAPI_GOODSMVT_CREATE with your payload. Review RETURN messages and correct the data accordingly. Ensure BAPI_TRANSACTION_COMMIT is called after successful execution.
User Query 3
"Getting Error as Interface Mapping Missing for Account AMEX122 for Business Unit: AMEX001, please add required interface mapping and reprocess the transaction from source system or from message monitor."
AI Response
This is a classic Data Setup problem related to GL Account mapping configuration. Root Cause: The middleware cannot find a corresponding SAP GL Account mapping for the source account "AMEX122" under Business Unit "AMEX001". This mapping is essential to let the middleware know how a GL Account is mapped from CTRM to SAP. Resolution Steps: 1. Login to the ERP-Connect Web UI 2. Navigate to the Interface Mapping widget 3. You will find existing mappings already defined there 4. Copy an existing record for GLAccount as a template 5. Create a new mapping entry: • Source Account: AMEX122 • Business Unit: AMEX001 • Target SAP GL Account: (e.g., 1234545 – confirm with Finance team) 6. Save the mapping configuration Final Step: Once the mapping is saved, reprocess the failed transaction from the Message Monitor. The transaction should now process successfully with the correct GL Account mapping in place.
